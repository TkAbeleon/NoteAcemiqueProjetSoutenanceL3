

  


  

**Pour chaque alerte importante**, tu enverras **2 images** (pas plus, pour éviter de surcharger) :

  

1. **Image 1 : Vue Thermique (Infrarouge)** → **Priorité absolue**

- Montre les zones chaudes (feux actifs ou sol brûlé)

- Très utile même avec des nuages

- Source principale : **Landsat 8/9 (B10/B11)** ou **VIIRS/MODIS TIR**

  

2. **Image 2 : Vue Visible / RGB** (ou RGB + NIR)

- Photo "naturelle" pour comprendre le contexte (forêt, savane, village)

- Source : **Sentinel-2** ou **Landsat 8/9**

  

**Option bonus (si possible)** : Une **3ème image** = Carte burned area (Copernicus ou MCD64A1) avec contour de la zone brûlée.

  

---

  

### Stratégie finale pour les alertes (recommandée)

  

| Niveau d’alerte | Images envoyées | Contenu du message | Fréquence |

|-----------------------|----------------------------------------------|--------------------------------------------------|---------|

| **Alerte Haute** (FRP élevé) | Thermique + Visible + Carte burned area | "Feu important détecté - 12 ha brûlés" + 3 liens/images | Prioritaire |

| **Alerte Moyenne** | Thermique + Visible | "Feu détecté - Estimation 4 ha" + 2 images | Normal |

| **Alerte Basse** | Seulement Thermique | "Point chaud détecté" + 1 image | Optionnel |

  

---

  

### Comment obtenir ces images (solution simple)

  

- **Automatisation** : Quand un cluster est détecté → ton workflow n8n lance un script Python qui :

1. Cherche la scène Landsat/Sentinel-2 la plus récente pour cette zone

2. Télécharge les bandes nécessaires

3. Génère 2-3 PNG (visible + thermique)

4. Upload sur **ImgBB** (

5. Récupère les liens directs

6. Envoie l’alerte avec les liens

  

**Temps estimé par alerte** : 30 à 90 secondes (acceptable pour un prototype).

  

---

  

### Avantages de cette décision

  

- Contourne bien les nuages (grâce à l’image thermique)

- Donne de la **valeur réelle** aux alertes (les gens voient vraiment le feu)

- Reste **réaliste** dans tes contraintes (pas de DL lourd)

- Impressionnant en soutenance

  

**✅ DÉCISIONS FINALES DE CONCEPTION – JeryMotro**

*(Version officielle – 23 février 2026)*

  

Voici le récapitulatif **complet, clair et final** de toutes les décisions architecturales et techniques prises pour ton projet de mémoire L3.

  

### 1. Objectif Général du Projet

Concevoir et développer une **plateforme intelligente de détection précoce et d’alerte des feux de brousse à Madagascar**, nommée **JeryMotro**, en se concentrant sur l’ingénierie logicielle tout en respectant des contraintes matérielles, financières et temporelles (3 mois, sans GPU puissant, sans budget).

  

### 2. Décisions Techniques Majeures

  

| Domaine | Décision Finale | Justification |

|---------------------------|------------------------------------------------------|-------------|

| **Source de données principale** | Données points CSV FIRMS (MODIS + VIIRS) | Léger, rapide, gratuit, temps réel |

| **Enrichissement des données** | Google Earth Engine (GEE) – NDVI, ERA5 (température, humidité, vent, précipitations), pente SRTM | Gratuit, puissant, sans GPU |

| **Estimation superficie brûlée** | Utilisation de produits prêts-à-l’emploi : MCD64A1 (NASA) + Copernicus Burnt Area 300m daily | Pas d’analyse pixel par pixel, faible complexité |

| **Images satellites** | Téléchargement sélectif (Landsat 8/9 + Sentinel-2) sur zones prioritaires (FRP élevé) :<br>• Bande Visible/RGB<br>• Bande Thermique (TIR) | Contourne les nuages + visuels pour alertes |

| **Modèle ML/DL** | **MadFireNet** :<br>• XGBoost (classification risque)<br>• ConvLSTM léger (prédiction J+1)<br>• Pas de U-Net ou DL lourd sur images | Respecte les contraintes GPU/temps |

| **Clustering** | HDBSCAN (spatio-temporel) | Identification des événements feu |

| **Automatisation** | **n8n** (Docker) | Orchestration complète du pipeline |

| **Frontend** | **React** | Choix final de l’étudiant |

| **Backend / Dashboard** | Streamlit (principal) + React (interface avancée) | Rapidité de développement + modernité |

| **Base de données** | Supabase (PostgreSQL) | Gratuit, simple, scalable |

| **Stockage images** | Google Drive (stockage) + ImgBB (liens rapides pour alertes) | Gratuit, liens directs |

| **Système d’alertes** | Email + WhatsApp/SMS via **carte SIM personnelle** (Pushbullet + MacroDroid) | Pas d’API opérateur, solution locale |

  

### 3. Architecture Finale du Système

  

```mermaid

FIRMS API (toutes 30 min)

↓ (n8n)

Enrichissement GEE (NDVI, ERA5, pente)

↓

Clustering HDBSCAN + Burned Area (MCD64A1/Copernicus)

↓

MadFireNet (XGBoost + ConvLSTM)

↓

Stockage (Supabase + Google Drive)

↓

Dashboard (React + Streamlit)

↓

IA Explicative (Grok + RAG)

↓

Alertes : Email + SMS/WhatsApp (avec images Visible + Thermique)

```

  

### 4. Décisions sur le Déploiement

  

- **Frontend React** → Vercel (gratuit)

- **Backend Streamlit** → Streamlit Cloud (gratuit)

- **n8n** → Render ou Docker local (gratuit)

- **Base de données** → Supabase (gratuit)

- **Nom de domaine** → Sous-domaine Vercel (`jerymotro.vercel.app`) ou domaine gratuit Freenom (.ml / .tk)

  

**Objectif** : Avoir une plateforme en ligne fonctionnelle avant la soutenance.

  

### 5. Stratégie Images pour Alertes (décision finale)

  

- **Toujours envoyer** :

- 1 image **Thermique** (prioritaire – contourne nuages)

- 1 image **Visible/RGB** (quand disponible)

- **Optionnel** : Carte burned area

- **Stockage** : ImgBB pour liens rapides dans SMS/WhatsApp

  

### 6. Contraintes Acceptées

  

- Pas d’analyse pixel par pixel / DL lourd sur images

- Pas d’API opérateur payante (utilisation carte SIM personnelle)

- Budget = 0 Ar

- Pas de GPU puissant
